## 三、关键技术决策

### 3.1 BGE Query Instruction(用 question 版)

按老师要求,使用 `"Represent this question for searching relevant passages: "` 作为 query instruction(之前用 `sentence` 版,本次按规范改为 `question` 版)。

**影响**:让 bge-en 对"question 风格"的 query 召回效果更好(官方推荐)。

### 3.2 静态词典 vs UMLS/MeSH

**选择**:先用静态手写词典(覆盖 ~80 条)

**理由**:
- 作业场景不需要工业级医学标准库
- UMLS 是付费的,MeSH 免费但需要 ETL pipeline
- 静态词典易读、易扩展、易在报告里展示

**未来**:如果扩展到医疗专业场景,应接 UMLS / MeSH(可在报告里写为"改进方向")。

### 3.3 中文支持的 3 个关键修复

#### 修复 1:PD-1 vs PD 子串误匹配

**问题**:`\bpd\b` 会匹配 "PD-1"(因为 D 和 - 之间是单词边界)

**解法**:**长 key 优先 + 位置覆盖** 算法
- 按 key 长度降序排(`pd-1` 4 字符先于 `pd` 2 字符)
- 维护 `covered` 集合跟踪已匹配位置
- 短 key 的 match 落在 covered 内就跳过

**效果**:Q5/Q6/Q7 中"PD-1" 都正确识别为免疫检查点(无帕金森误触发)。

#### 修复 2:中文没有 \b 单词边界

**问题**:`\b` 只对 ASCII 起作用,中文是连续字符流

**解法**:在 `_make_pattern()` 中检测 key 字符类型
- 英文 key → `\b` 单词边界
- 中文 key → **子串包含即匹配**(`in` 操作符)

**理由**:中文"词边界"不严格,子串误匹配反而少(且"心脏"匹配"心脏搭桥"是合理的医学召回)。

#### 修复 3:中文 regex 字符类拆词失败

**问题**:`近[三5五]年` 用 `_extract_cn_candidates` 抽候选词时,字符类 `[三5五]` 被拆掉,只留下 "近" 和 "年" 两个单字符(被长度过滤掉)

**解法**:**展开字符类**
- `近[三5五]年` → `近三年|近五年`
- `过去[三5五]年` → `过去三年|过去五年`
- `过去[一1]年` → `过去一年|过去1年`

**效果**:Q7 "近五年..." 的时间过滤正确提取为 `year≥2021`。

### 3.4 增强后的 Query 结构

```json
{
  "original": "心梗病人能不能用阿司匹林?",
  "cleaned": "心梗病人能不能用阿司匹林?",
  "entities": {"drug_cn": ["阿司匹林"]},
  "synonyms": ["aspirin", "心肌梗死", "心肌梗塞", "myocardial infarction", "乙酰水杨酸"],
  "query_variants": [原始 + 5 个同义词追加版,共 6 个],
  "vector_query": "Represent this question for searching relevant passages: 心梗病人能不能用阿司匹林?",
  "keyword_query": "心梗病人能不能用阿司匹林? aspirin 心肌梗死 心肌梗塞 myocardial infarction 乙酰水杨酸",
  "filter_conditions": {},
  "enhancement_log": [...]
}
```

**给下游 RAG 的接口**:
- `vector_query` → 喂给 bge embedding
- `keyword_query` → 喂给 BM25 (阶段六/七的 hybrid)
- `query_variants` → 多版本召回(RRF 融合)
- `filter_conditions` → 喂给 ChromaDB where 参数

## 四、踩坑记录(技术深度)

### 4.1 ChromaDB 0.5 之前在阶段四踩过

- 不支持 `$contains` 等子串匹配
- 单 dict 只能一个 operator,多条件用 `$and`
- 本阶段没新增 chroma 相关问题

### 4.2 本周 3 个新坑

| 坑 | 现象 | 根因 | 修复 |
|----|------|------|------|
| PD-1 误触发 PD | "PD-1 immunotherapy" 返回帕金森同义词 | `\bpd\b` 在 "PD-1" 中合法匹配 | 长 key 优先 + 位置覆盖 |
| 中文 lookahead 误杀 | "二甲双胍" 后面是 "对",`(?![一-龥])` 拒绝 | 汉字边界规则不适用中文 | 中文用 `in` 检查 |
| regex 字符类拆词 | `近[三5五]年` 拆词得到 "近" "年" | `_extract_cn_candidates` 只抽连续汉字 | 展开字符类 |

### 4.3 一个设计决策的反思

最初为中文 entity 模式写的是 negative lookahead `(?<![一-龥])...(?![一-龥])`,理论上"前后是汉字就不匹配"。但**中文没有真正的"词边界"** — "二甲双胍对..." 中 "二甲双胍" 是独立词,后面"对"是另一词的开头,从字符角度它们相邻。

**反思**:在中文 NLP 任务中,使用 negative lookahead/lookbehind 来定义"词边界"是个**反模式**。要么用词典(contains),要么用专业分词工具(jieba、HanLP)。

## 五、已知问题与限制

| 限制 | 影响 | 应对 |
|------|------|------|
| **静态词典覆盖不全** | 长尾医学术语无法识别 | 后续接 UMLS / MeSH |

## 八、心得体会

### 8.1 医学 RAG 检索的关键挑战

本周最深的体会:**医学领域的检索增强,核心不在 embedding 模型,而在"医学知识"**。

- 同样的 query,`bge-en` vs `bge-zh` vs `bge-large` 的差异 < 5%
- 但有没有同义词扩展、有没有实体识别、有没有时间过滤,差异可能有 **30-50%**

这就是为什么 **query 增强** 是 RAG 系统里 ROI 最高的优化点。

### 8.2 中英文混合是真实场景的常态

**不要假设 query 一定是英文**(即使数据是英文文献)。用户:
- 医生查房时输入中文术语
- 论文写作时混用 PD-1、EGFR 等缩写
- 跨语言搜文献是真实需求

所以**中英文双语 query 增强是必备能力**,不是 nice-to-have。

### 8.3 静态词典的局限与替代方案

| 方案 | 优 | 劣 |
|------|----|----|
| 静态手写词典 | 易读、易控、零依赖 | 覆盖不全、难维护 |
| **UMLS / MeSH** | 工业级、覆盖 1000w+ 概念 | 需 ETL pipeline、有授权问题 |
| **LLM 提取** | 覆盖无限、零维护 | 慢、贵、可能幻觉 |
| **专用 NER 模型** | 快、准 | 需训练/微调 |

作业场景:静态词典足够(展示原理 + 80 条核心);
生产场景:**UMLS / MeSH + LLM 提取**是更稳的方案。
