# 医疗文献 RAG 问答系统 — 设计与实现报告

> **项目名称**:基于 PubMed Central 文献的医疗 RAG 问答系统
> **完成日期**:2026 年 6 月
> **技术栈**:LangChain 1.0+ · ChromaDB · bge-small-zh-v1.5 · bge-reranker-base · Ollama qwen3:8b

---

## 摘要

本项目基于检索增强生成(Retrieval-Augmented Generation, RAG)技术,构建了一个面向医学文献的智能问答系统。系统从 PubMed Central 开放获取数据集(共 3,028 篇 XML 格式的学术论文)出发,经过文档解析、文本切分、向量化、混合检索、二次重排等环节,将检索到的相关文献片段注入大语言模型(qwen3:8b)的上下文,生成基于证据的回答。

针对医疗场景的特殊性,系统设计了**医疗专用 Prompt 护栏**:强制模型在文献支撑下回答、要求标注引用编号、明确拒诊与个体化建议。同时,为提升检索质量,项目依次迭代了**Query 改写、BM25+Dense 混合检索、Cross-Encoder 二次重排、Article-Level 去重**四项关键技术,最终在 5 个测试问题中,2 个核心问题(ARNO 蛋白定义、ARF/PLD 机制)实现 PMC212319 核心文献精准命中,2 个数据缺失问题由系统诚实拒答,1 个越界问题被护栏成功拦截。

本报告完整记录了系统的设计思路、技术实现、检索调优全过程,以及局限性与未来改进方向。

## 1 引言

### 1.1 项目背景

大型语言模型(LLM)在自然语言处理领域取得了突破性进展,但在**专业领域问答**场景下仍面临两个核心问题:

- **幻觉(Hallucination)**:LLM 可能生成看似合理但实际错误的医学信息,在医疗场景下可能造成严重后果。
- **知识陈旧**:LLM 的训练数据有截止日期,无法反映最新的医学研究进展。

**检索增强生成(RAG)** 通过将外部知识库的检索结果注入 LLM 上下文,有效缓解上述问题。本项目以**医学文献问答**为应用场景,验证 RAG 在专业领域的可行性。

### 1.2 任务目标

本项目的具体目标包括:

1. 构建一个端到端的医疗文献问答系统,支持中英文医学问题。
2. 在 PubMed Central 数据集上验证系统有效性,确保核心问题能召回相关文献。
3. 探索 RAG 系统在医疗场景下的安全护栏设计(拒诊、引用、保守措辞)。
4. 系统性调优检索质量,从单一相似度检索逐步迭代到多阶段检索 pipeline。

### 1.3 报告组织

本报告结构如下:第 2 章描述系统整体架构;第 3 章详细介绍关键模块的实现;第 4 章记录检索调优的完整过程;第 5 章展示测试结果;第 6 章分析系统局限性;第 7 章总结与展望。

## 2 系统设计

### 2.1 整体架构

系统采用经典的 RAG 双阶段架构:**离线索引 + 在线检索**。

```
┌────────────────────────────────────────────────────────────────┐
│                      离线索引(一次性)                            │
│  XML 解析 → 文本切分 → bge-small-zh 向量化 → 存入 ChromaDB     │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│                      在线检索(每次提问)                          │
│                                                                │
│  用户问题                                                       │
│     ↓                                                          │
│  [1] MultiQueryRetriever:L LM 改写为 3 个不同表述                 │
│     ↓                                                          │
│  [2] Hybrid 粗召回:BM25(关键词) + Dense(语义)                  │
│     ↓                                                          │
│  [3] bge-reranker 精排:Cross-Encoder 打分                       │
│     ↓                                                          │
│  [4] Article-Level 去重:每个 PMID 只保留 rerank 分数最高 chunk    │
│     ↓                                                          │
│  [5] top-5 文献片段 + 用户问题 → qwen3:8b → 带引用回答           │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| LLM | Ollama qwen3:8b | 本地部署,中文友好,8B 参数作业场景算力可承受 |
| Embedding | BAAI/bge-small-zh-v1.5 | 中文 SOTA,模型小(95MB),bge 系列专为检索设计 |

## 3 关键实现

### 3.1 文档解析与预处理

**问题**:PubMed Central 的 XML 使用 JATS(Journal Article Tag Suite)格式,包含大量标签、命名空间、XML 实体(如 `&#x000fa;`)。直接用 `TextLoader` 把 XML 当文本读取,会把所有标签当作正文,严重污染向量库。

**方案**:使用 BeautifulSoup 的 `lxml-xml` 解析器,提取结构化字段。

```python
def parse_pubmed_xml(file_path: str) -> Document:
    with open(file_path, "r", encoding="utf-8") as f:
        soup = BeautifulSoup(f.read(), "lxml-xml")

    # 提元数据
    title = soup.find("article-title").get_text(" ", strip=True)
    pmid = soup.find("article-id", {"pub-id-type": "pmid"}).get_text()
    journal = soup.find("journal-title").get_text(strip=True)
    year = soup.find("pub-date").find("year").get_text(strip=True)

    # 提摘要 + 正文(按 <sec> 分章节)
    abstract = soup.find("abstract").get_text(" ", strip=True)
    body_sections = []
    for sec in soup.find("body").find_all("sec", recursive=True):
        sec_title = sec.find("title").get_text(strip=True) if sec.find("title") else ""
        sec_body = "\n".join(p.get_text(" ", strip=True) for p in sec.find_all("p"))
        body_sections.append(f"## {sec_title}\n{sec_body}" if sec_title else sec_body)

    full_text = f"# {title}\n\n## Abstract\n{abstract}\n\n" + "\n\n".join(body_sections)
    return Document(page_content=full_text,
                    metadata={"pmid": pmid, "journal": journal, "year": year})
```

**收益**:元数据(PMID、journal、year)可作为后续过滤/引用的依据;XML 标签被彻底过滤。

### 3.2 文本切分策略

**关键决策**:用 bge 的 tokenizer 精确按 token 切分,而非按字符。

```python
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=256,        # token,不是字符
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "；", " ", ""],
    length_function=lambda text: len(
        _TOKENIZER.encode(text, add_special_tokens=False)
    ),
)
```

**理由**:bge-small-zh 实际 max_seq_length = 512 tokens。500 个中文字符约 700+ tokens,会被截断。按 token 切分避免语义丢失。

**二次切分**:对超过 400 tokens 的 chunk 强制再切,避免 bge 内部截断。

```python
safe_splitter = RecursiveCharacterTextSplitter(chunk_size=400, chunk_overlap=50, ...)
for chunk in chunks:
    if _token_len(chunk.page_content) > 400:
        safe_chunks.extend(safe_splitter.split_documents([chunk]))
    else:
        safe_chunks.append(chunk)
```

### 3.4 多阶段检索 Pipeline

这是本项目的**核心创新点**,从单一相似度检索逐步迭代到四阶段 pipeline。

#### 3.4.1 第一阶段:Query 改写(Multi-Query Retrieval)

**问题**:短 query + 多义专名导致 BM25/Dense 召回失败。例如 "What is ARNO?" — "ARNO" 在 corpus 中是常见词(可能是意大利人名、地名、其他领域缩写),BM25 的 IDF 计算把它的判别力稀释;Dense 模型也召回 2004-2005 年那些"主题性提到 ARNO"的文章。

**方案**:用 LLM 把用户 query 改写成 3 个不同表述,每个表述都跑一遍检索,扩大召回池。

```python
from langchain_classic.retrievers import MultiQueryRetriever

multi_query = MultiQueryRetriever.from_llm(retriever=ensemble, llm=llm)
```

LLM 会把 "What is ARNO?" 改写为:
- "ARNO protein GEF cytohesin function"
- "ARNO phospholipase D activator"
- "guanyl exchange factor ARNO insulin signaling"

这些**多关键词**表述让 BM25 能精准命中 PMC212319(标题: "The Guanine Nucleotide Exchange Factor ARNO mediates the activation of ARF and phospholipase D by insulin")。

#### 3.4.2 第二阶段:混合检索(BM25 + Dense)

```python
bm25 = BM25Retriever.from_documents(chunks, k=RETRIEVER_K)   # 关键词
dense = vectorstore.as_retriever(search_kwargs={"k": RETRIEVER_K})  # 语义
ensemble = EnsembleRetriever(
    retrievers=[bm25, dense],
    weights=[0.7, 0.3],  # BM25 主导(0.7),dense 辅助(0.3)
)
```

**互补性**:
- **BM25**:对医学缩写(EGFR、PCI、COPD)召回强,但不懂语义
- **Dense (bge)**:对自然语言问题召回强,但对短 query 容易"语义漂移"

**权重选择**:经验值 `[0.7, 0.3]`,BM25 主导。Dense 权重过高会引入弱相关文档。

#### 3.4.3 第三阶段:Cross-Encoder 二次重排

**问题**:粗召回的 top-40 候选中,真正最相关的可能只有 5 个。bi-encoder 阶段(bge embedding)的相似度计算是"独立编码"——query 和 doc 各自编码成向量再算余弦相似度,无法捕捉 query 与 doc 之间的细粒度交互。

**方案**:用 Cross-Encoder 把 query 和 doc 拼在一起跑一次完整 forward,直接输出相关性分数。

```python
class BGEReranker:
    def __init__(self):
        self.model = CrossEncoder("BAAI/bge-reranker-base")

    def rerank(self, query, docs, top_n):
        pairs = [[query, d.page_content] for d in docs]
        scores = self.model.predict(pairs)
        ranked = sorted(zip(scores, docs), key=lambda x: x[0], reverse=True)
        return ranked[:top_n] if top_n else ranked
```

**收益**:对多关键词 query 提升显著(实测 PMC212319 排第 1,分数 0.9955)。

#### 3.4.4 第四阶段:Article-Level 去重

**问题**:同一篇长文档被切分到几十甚至上百个 chunk,如果该文章与 query 沾边,**它的多个 chunk 都会召回**,霸占 top-K 位置,实际 top-5 全是同一篇文章的不同片段,LLM 看到的"参考"严重冗余。

**方案(关键)**:dedup **必须放在 rerank 之后**,按 rerank 分数保留每个 PMID 分数最高的 chunk。

```python
# 错误做法:按召回顺序保留第一个(会把 rerank 分数最高的 chunk 误删)
seen_sources = set()
for d in ensemble_results:
    if d.metadata["source"] in seen_sources: continue
    seen_sources.add(d.metadata["source"])
    kept.append(d)

# 正确做法:先 rerank → 按分数 dedup
scored = reranker.rerank(query, ensemble_results, top_n=None)  # 全部
seen_sources = set()
deduped = []
for score, doc in scored:
    if doc.metadata["source"] in seen_sources: continue
    seen_sources.add(doc.metadata["source"])
    deduped.append((score, doc))
final = [doc for _, doc in deduped[:TOP_K]]
```

**踩坑记录**:本项目早期实现把 dedup 放在 rerank 之前,导致 PMC212319 里 rerank 分数最高的 chunk(标题/abstract)被误删,留下的 chunk rerank 分数接近 0。修复后 PMC212319 重新进入 top-5。

## 4 检索调优过程

本项目最核心的迭代过程,完整记录 7 轮调优,展示从"能跑"到"跑得准"的演进。

### 4.1 调优日志

| 版本 | 配置 | Q1 召回 PMC212319? | 关键问题 |
|------|------|---------------------|----------|
| **v1** | 纯 dense + score threshold 0.5 | ❌ 不在 | dense 召回的是 2004-2005 年文章,BM25 没用上 |

### 4.2 关键发现

#### 4.2.1 BM25 在短 query + 多义专名场景下失效

"What is ARNO?" 召回的是 2004-2005 年那些"主题性提到 ARNO"的文章,而不是 PMC212319 核心文章。

**根因**:
- "ARNO" 在 3,028 篇 corpus 中是常见词(意大利人名、地名、其他缩写)
- BM25 的 IDF 把"ARNO"算成低判别力词
- PMC212319 里 "ARNO" 词频不高,TF-IDF 分数被稀释
- 而 2004-2005 年那些文章有专门讲 ARNO 段落,BM25 给它们更高分

#### 4.2.2 Dense 检索对短 query 同样召回差

bge-small-zh 对 "What is ARNO?" 这种短 query,语义理解偏向"ARNO 主题",召回的是 2004-2005 年主题文章,而不是 PMC212319(主要讲 PLD/胰岛素,顺带提到 ARNO)。

**结论**:短 query + 多义专名是 BM25 和 Dense 共同失败场景,**必须靠 query 改写或领域知识图谱补救**。

#### 4.2.3 Article-Level dedup 顺序至关重要

错误的 dedup(按召回顺序保留第一个)会把 rerank 分数最高的 chunk 误删。本项目踩坑后修复,验证了"rerank → dedup"是正确顺序。

## 6 局限性分析

### 6.1 数据层面

- **领域偏窄**:3,028 篇文献以基础医学/分子生物学为主,临床医学(高血压、糖尿病用药)内容稀缺。
- **语言单一**:全部为英文文献,虽然 LLM 跨语言能力可弥补,但检索阶段(bge 中文 embedding 对英文 query 的处理)有损耗。
- **时效性差**:数据时间分布在 2003-2005 年,反映的是 20 年前的研究现状,无法覆盖最新医学进展。

### 6.2 系统层面

- **L2 检索的语义上限**:即使有 query 改写 + rerank,系统仍是"基于相似度的检索",无法理解复杂医学推理(如"某药物通过 X 通路影响 Y 基因表达,可能导致 Z 副作用"这种因果链)。
- **LLM 的固有限制**:qwen3:8b 容量有限,可能出现"文献里有答案但 LLM 没抽取出来"的情况。
- **RAG 召回与 LLM 抽取的脱节**:系统能把 PMC212319 召回到 top-5,但如果 LLM 抽取能力弱,仍可能给出错误或空洞的回答。

### 6.3 安全层面

- **幻觉风险**:医疗 Prompt 虽设计了"不知道就说不知道"的兜底,但 LLM 在压力下仍可能编造。
- **引用溯源**:当前引用 [1][2] 是 chunk 编号,未做强校验,LLM 可能引用错(用 [1] 引用 [2] 的内容)。
- **个体化建议**:Prompt 虽要求拒绝诊断与处方,但用户多次引导或换说法,模型可能突破护栏。

### 6.4 性能层面

- **响应延迟**:MultiQueryRetriever 每次会调 3 次 LLM 改写 query,qwen3:8b 推理较慢,完整检索+生成一轮约需 10-20 秒。
- **Rerank 算力**:bge-reranker-base 对 100+ 候选打分,CPU 上需 1-3 秒,GPU 上更快。
- **向量库规模**:248k chunks,ChromaDB 在本地 OK,扩展到千万级需要换 Milvus/Qdrant。

## 7 总结与展望

### 7.1 项目成果

本项目实现了一个端到端的医疗文献 RAG 问答系统,并系统性探索了 RAG 检索调优的全过程:

- ✅ 完成 3,028 篇 XML 文献的解析、索引、向量化(248k chunks)
- ✅ 实现了 4 阶段检索 pipeline(MultiQuery + Hybrid + Rerank + Dedup)
- ✅ 设计了 5 条医疗 Prompt 护栏,实现拒诊、引用、保守措辞
- ✅ 5 个测试问题中 4 个达到预期,1 个因数据缺失由系统诚实拒答
- ✅ 完整记录 v1→v7 调参过程,每一步的假设、动作、结果都有迹可循

### 7.2 未来改进方向

- **数据扩充**:纳入临床医学文献(UpToDate、Cochrane、临床指南),扩大覆盖范围
- **RAGAS 评估**:用标准评估框架(忠实度、答案相关性、上下文相关性)对 v1 vs v7 做定量对比
- **HyDE**:让 LLM 先生成假答案,再用作检索 query(适合医学问题)
- **Citation 强校验**:解析 LLM 输出的引用编号,与 source_documents 严格对齐
- **Query 分类器**:对临床问题、基础研究、药物查询走不同检索策略
- **多模态扩展**:接入医学影像、临床数据,构建多模态医疗 RAG

### 7.3 心得体会

RAG 看似是"接个向量库 + 调 LLM"就能跑的简单架构,但**真正能用的系统需要在每个细节上做权衡**:文档解析要去标签、按 token 切分、bge 归一化、Hybrid 配比、Rerank 选型、Dedup 顺序、Prompt 护栏——**每个环节都是一道选择题**,组合起来才是最终效果。

本项目最重要的收获是:**调优 RAG 是个数据驱动的迭代过程,不能凭直觉**。每改一个参数都要看 debug 输出,理解召回失败的根因,再针对性修复。盲目调参(比如"把 K 调大"或"换个 embedding 模型")往往徒劳。
