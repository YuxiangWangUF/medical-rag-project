### 3.2 ⚠️ 已知问题 1:首块"标题孤立 chunk"

**现象**:
```
chunk_id: 12929205_chunk_000
text: "The Transcriptome of the Intraerythrocytic Developmental Cycle of Plasmodium falciparum"
token_count: 14        ← 极短
```

**每个被分割的文献,chunk_000 几乎都是"只有标题"**,只有 14 token(标题字符数)。

**根因**:
- 文本拼接方式为 `f"{title}\n\n{abstract}\n\n{body_text}"`
- `RecursiveCharacterTextSplitter` 第一优先级 separator 是 `\n\n`
- 标题是一个独立段落(< 256 token),被识别为完整 chunk
- 后续 abstract + body 一起被切分成多个 chunk

**影响**:
- 标题 chunk 信息量低,被检索时召回概率低
- 但**作为"主题标签"反而有用**:LLM 看到 chunk_001 时,能回溯到 chunk_000 的标题
- 实际上,这个行为**接近 RAGAS 推荐的"标题注入"模式**(每篇文档首块放标题)

**建议**:
- ✅ **可接受,无需修改**
- 或者:把 `separators` 里的 `"\n\n"` 改为只在 abstract/body 之间使用,标题和 abstract 之间不强制分隔

### 3.3 ⚠️ 已知问题 2:末尾正常结束仅 6.21%

**现象**:
- 138,142 chunk 中,只有 8,584 个(6.21%)末尾是标点或换行
- 大部分 chunk 在 token 边界处被切断(不在句末)

**根因**:
- `RecursiveCharacterTextSplitter` 按 token 数切分,不在乎句末
- 这是 token-based 切分的固有 trade-off

**影响**:
- 对**检索质量**影响小(bge embedding 不在乎句末)
- 对**人工阅读 chunk**有损(看到的 chunk 可能不完整)
- 对 LLM **回答质量**影响小(LLM 能从上下文推断)

**建议**:
- ✅ **可接受,已是合理 trade-off**
- 如果追求完美,可改用 sentence-based 切分(sentence-transformers 库),但召回质量可能下降

### 3.4 ❌ 已知问题 3:重叠检查方法有 bug

**现象**:
- 报告说"样本 doc 重叠 0 字符"
- 但 `chunk_overlap=50` 应该有 ~50 token 重叠

**根因**:
- 当前检查方法用"找最长公共前缀字符"(chunk1 末尾 200 字符 vs chunk2 开头 200 字符逐字符对比)
- 这种方法对**单字符级差异敏感**,separator 和空格差异会让"公共前缀 = 0"
- 实际 overlap 存在,只是检测方法太严格

**修复**(已记入后续):
- 改用 token 级别 overlap 计算:`chunk1 最后 N token 文本 vs chunk2 开头 N token 文本,看 N 是否接近 chunk_overlap`
- 或用 `diff.SequenceMatcher` 算相似度

**实际 overlap 估算**:
- chunk_size=256, overlap=50 → overlap 比例 19.5%
- 期望每个 chunk 与下一 chunk 有 50 token 重复内容
